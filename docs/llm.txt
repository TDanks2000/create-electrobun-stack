create-electrobun-stack CLI usage guide for LLMs

Use this file when helping a user scaffold an Electrobun desktop app with create-electrobun-stack.

Primary command:

bunx create-electrobun-stack my-app

Local development command inside the create-electrobun-stack repository:

bun run src/index.ts my-app

What the CLI creates:
- An Electrobun desktop app.
- Bun main process entrypoint.
- React renderer in an Electrobun WebView.
- Electrobun Edit menu by default.
- Local-only renderer navigation guard by default.
- TanStack Router.
- Tailwind CSS by default, or plain CSS when requested.
- Strict TypeScript.
- Biome.
- Bun test by default.
- Typed native Electrobun RPC bridge.
- ces.json in the generated project root with Better-T-Stack-style stack metadata, a reproducible command, and feature flags.

Default behavior:
- If the user provides only a project name, scaffold the default stack.
- Dependencies are installed with Bun by default.
- Git is not initialized by default.
- The current implemented template is minimal.
- SQLite is implemented for the minimal template.
- Drizzle is implemented when paired with --database sqlite.
- --api none is implemented and omits BrowserWindow RPC wiring.
- --auth app-lock is implemented as a local app-name lock screen.
- --db-setup seed is implemented when paired with --database sqlite.
- --addons turborepo is implemented.
- --package-manager supports bun, npm, pnpm, and yarn for dependency installation.
- The default --app-menu edit adds an Electrobun ApplicationMenu with standard Edit roles.
- --app-menu none omits the generated menu file and setup call.
- The default --navigation local-only blocks non-views:// navigation from the renderer.
- --navigation none omits the will-navigate guard.
- --window-style hidden-inset adds Electrobun hiddenInset titlebar config and a draggable header.
- --build-env controls electrobun build --env.
- --build-targets controls electrobun build --targets.
- The default --testing bun renders a Bun test script and generated manifest smoke test.
- --testing none omits the test script and test files.
- The default --examples rpc renders a starter RPC greeting/logging demo.
- --examples none keeps the RPC bridge but omits the demo calls and generated README section.
- Every generated project includes ces.json for future create-electrobun-stack commands.
- ces.json has top-level fields like version, createdAt, reproducibleCommand, database, orm, frontend, addons, examples, api, and testing.

Recommended commands:

Create a default app:

bunx create-electrobun-stack my-app

Create without installing dependencies:

bunx create-electrobun-stack my-app --no-install

Create and initialize git:

bunx create-electrobun-stack my-app --git

Preview without writing files:

bunx create-electrobun-stack my-app --dry-run

Create in another parent directory:

bunx create-electrobun-stack my-app --cwd ~/Desktop

Override the Electrobun app identifier:

bunx create-electrobun-stack my-app --app-id com.example.myapp

Use non-interactive defaults:

bunx create-electrobun-stack my-app --yes

List templates:

bunx create-electrobun-stack --list-templates

Print help:

bunx create-electrobun-stack --help

Print version:

bunx create-electrobun-stack --version

Important current limitations:
- Only the minimal template is implemented.
- standard and full are implemented profiles that currently share the minimal template source.
- Invalid combinations fail before scaffolding unless --dry-run is used.

Generated app commands after scaffolding:

cd my-app
bun run dev
bun run build
bun test
bun run typecheck
bun run lint
bun run format

If the user created with --no-install, run:

cd my-app
bun install
bun run dev

Behavior notes:
- If no project name is provided in an interactive terminal, the CLI prompts for one.
- If no project name is provided with --yes or in non-interactive mode, the CLI fails.
- Empty existing target directories are allowed.
- Non-empty target directories fail before files are written.
- --dry-run is safe and writes no files.
- --app-id must be reverse-DNS style, for example com.example.myapp.

Electrobun notes for answers:
- The starter uses Electrobun native typed RPC, not custom IPC.
- The Bun side uses BrowserView.defineRPC.
- The WebView side uses Electroview.defineRPC and new Electroview({ rpc }).
- The app loads the renderer through views://.
- Do not invent Electrobun APIs when explaining or extending the generated app.
